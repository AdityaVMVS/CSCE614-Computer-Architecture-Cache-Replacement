## Project Overview

Efficient cache management is a critical problem in multicore processor design. The shared last-level cache (LLC) must decide which blocks to retain or evict to balance:

- Hit rate / MPKI  
- Performance (IPC, speedup)  
- Hardware complexity and overhead  

In this project, we study a range of LLC replacement strategies, from classical policies (LRU, LFU, Rand) to more advanced predictive schemes (PACIPV, SRRIP, DRRIP, SHiP), using SPEC CPU2006 and PARSEC workloads in ZSim.

The ZSim codebase was modified by adding or updating the following files: init.cpp, drrip_repl.h, pacipv_repl.h, ship_repl.h.

---

## Project Goals

- Implement **PACIPV** (Prefetch-Aware Coarse-grained Insertion and Promotion Vectors) in ZSim.
- Compare PACIPV against a set of baseline and advanced cache replacement policies:
  - LRU  
  - LFU  
  - NRU  
  - Rand  
  - TreeLRU  
  - SRRIP  
  - DRRIP  
  - SHiP  
  - (Mockingjay planned for comparison but not implemented due to constraints)
- Collect detailed simulation data (cycles, instructions, misses, MPKI, IPC, weighted speedup).
- Analyze differences in behavior across SPEC (single-core) and PARSEC (8-core) workloads.
- Document implementation tradeoffs, limitations, and observations.

## Implemented Replacement Policies

The following LLC replacement policies are implemented or extended in this repository:

- **PACIPV** вЂ” Prefetch-Aware Coarse-grained Insertion and Promotion Vectors (demand-only variant in our setup)
- **SRRIP** вЂ” Static Re-Reference Interval Prediction
- **DRRIP** вЂ” Dynamic Re-Reference Interval Prediction
- **SHiP** вЂ” Signature-based Hit Predictor (with address-region signatures)
- **LRU** вЂ” Least Recently Used
- **LFU** вЂ” Least Frequently Used
- **NRU** вЂ” Not Recently Used
- **Rand** вЂ” Random replacement
- **TreeLRU** вЂ” Tree-based pseudo-LRU

Each policy is integrated with ZSimвЂ™s replacement framework and can be selected via the LLC replacement policy configuration.

  - Benchmarks: `bzip2`, `calculix`, `cactusADM`, `gcc`, `hmmer`, `lbm`, `libquantum`, `mcf`, `namd`, `sjeng`, `soplex`, `xalan`

The LLC is configured as a shared 16 MB last-level cache. Other cache and core parameters follow the courseвЂ™s ZSim configuration templates.

Simulation runs are organized per policy and per benchmark, with outputs stored in structured directories.

## Analysis and Metrics

All analysis is performed using Python scripts included in the repository.

From each `zsim.out`, we extract:

- Per-core cycles  
- Contention cycles  
- Retired instructions  
- LLC misses  

From these, we compute:

- **IPC** (Instructions Per Cycle)  
- **MPKI** (LLC Misses Per Kilo-Instruction)  
- **Weighted Speedup (WS)** vs. LRU  
- **Geometric-mean IPC** for:
  - All workloads
  - SPEC only
  - PARSEC only  
- **Average MPKI** for SPEC and PARSEC

The analysis pipeline generates:

- CSV summary files (per policy and benchmark group)
- Bar charts and summary plots for IPC, MPKI, and weighted speedup
- A consolidated summary table for all policies

These are used directly in the final project report.

## Limitations

Due to constraints in our ZSim environment and available interfaces, the following limitations apply:

* **PACIPV prefetch integration disabled**

  * Attempts to enable prefetch-aware behavior caused assertion failures in ZSimвЂ™s L3 prefetch path.
  * Our PACIPV implementation therefore operates as a **demand-only** variant and does not use separate demand/prefetch insertion and promotion vectors.

* **SHiP signatures approximate PC-based behavior**

  * ZSim does not expose the program counter (PC) in memory requests.
  * SHiP signatures are approximated using address-region hashing plus core identifiers (e.g., 16 KB regions), rather than true PC-based signatures.
  * This reduces the granularity of the Signature History Counter Table (SHCT) and deviates from the original SHiP design.

* **Demand-only learning for both PACIPV and SHiP**

  * Prefetch metadata is not used in the learning mechanisms.
  * Both policies train only on demand GETS/GETX requests.

These limitations affect the fidelity of PACIPV and SHiP relative to the algorithms described in the original research papers and help explain why our measured results show behavior closely clustered around RRIP/LRU baselines.
